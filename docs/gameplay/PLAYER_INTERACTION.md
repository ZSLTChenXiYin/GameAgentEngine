# 玩家交互总览

本页收口玩家自然语言输入、结构化交互、群聊模式和游戏侧权威校验的主设计。

## 1. 总原则

- 玩家应作为 Engine 中的正式节点存在
- 玩家节点用于承载身份、关系、记忆和叙事位置
- 玩家节点不拥有高频真值状态
- 高频真值状态必须保留在游戏侧权威数据中

## 2. 自然语言输入的处理方式

当前推荐做法不是让自然语言直接修改游戏真值，而是：

1. 将输入解释为玩家意图
2. 抽取目标、场景、对象、前置条件和潜在风险
3. 对需要真值校验的部分回到游戏侧查询
4. 只有通过游戏侧规则验证的动作，才能真正落地
5. 再由 Engine 为 NPC 反应、群聊氛围和叙事反馈提供推理

## 3. Engine 与游戏侧的职责边界

- Engine：世界建模、NPC 理解、叙事组织、关系和记忆推理
- 游戏侧：HP、背包、金钱、任务阶段、场景占用、即时状态等高频真值

权威边界详见：

- [游戏状态权威边界](./GAME_STATE_AUTHORITY.md)

## 4. 推荐交互模式

当前交互建议至少区分这些模式：

- `direct_dialogue`：玩家与单 NPC 对话
- `group_chat`：玩家与一组 NPC 共享同一轮发言空间
- `gift_response`：玩家送礼或展示物品后触发 NPC 反应
- 其他高风险行为：先由游戏侧规则层校验，再决定是否进入 Engine 反应推理

## 5. play 模式映射

当前 `GameAgentWorker play` 中，REPL 命令大致可映射为：

- `/talk <npc>`：建立直接对话目标
- `/say <message>`：向当前会话或群体发言
- `/ask <npc> <message>`：面向指定对象发起发言
- `/gift <npc> <item>`：先由游戏侧执行赠礼，再请求 Engine 生成反馈
- `/show_item <npc> <item>`：先校验物品存在，再请求目标反应

## 6. 当前限制

- 群聊模式当前仍以单主响应者为主，不是多 NPC 并行推理
- 高风险自然语言动作不能直接绕过游戏侧规则层
- 玩家自由叙述适合做“意图解释输入”，不适合直接当成真值写入
