# 项目路线图

**中文** | [**English**](./ROADMAP_EN.md)

更新时间：2026-07-08

---

## 1. 文档目的

这份路线图用于说明 GameAgentEngine 当前所处阶段、后续的优先开发方向，以及 Engine、DevCli、Creator、SDK 四条产品线的同步原则。

路线图以 `Engine` 为主线：

- `Engine` 先定义真实能力边界
- `DevCli` 和 `Creator` 负责把这些能力暴露给开发者
- `SDK` 负责把这些能力封装成稳定的集成接口
- 文档、示例、打包产物随后跟进，避免信息漂移

---

## 2. 当前项目定位

GameAgentEngine 是一个位于游戏逻辑与大模型能力之间的运行引擎，目标是为游戏开发者提供：

- 基于节点、组件、记忆、关系的世界建模能力
- LLM 驱动的 NPC 推理、对话和世界推进能力
- 可控的运行时动作系统
- 可观测、可调试、可集成的开发工具链

配套工具职责如下：

- `Engine`：推理、状态演化、持久化、策略、日志、连续性、世界 tick
- `DevCli`：面向命令行和 AI 协作场景的开发工具
- `Creator`：面向图形界面的世界查看、编辑、排障工具
- `SDK`：面向游戏或服务端集成的 Go 调用封装

---

## 3. 当前基线能力

### Engine

当前已具备：

- 世界图模型：节点、组件、记忆、关系
- 三档推理管线：`vertical`、`polling`、`full`
- `world_tick`、`event_impact`、`scope_advance`、`timeline_replan`
- 世界设置、世界策略、待审批计划
- 世界工作副本、存档快照、快照恢复
- 连续性状态组件：`world_state`、`story_state`、`story_history`、`tick_policy`、`state_snapshot`
- 时间线归档、结构化日志、调试轨迹

### DevCli

当前已具备：

- 世界导入与校验
- 节点 / 组件 / 记忆 / 关系 CRUD
- 世界推进、事件评估、局部推进、时间线重规划
- 世界 rename、fork、snapshot、restore
- 日志、连续性、状态组件、时间线相关读取能力

### Creator

当前已具备：

- 世界与节点可视化编辑
- 节点复制、关系编辑、拖拽改父子关系
- 世界设置、世界策略、待审批计划查看
- `Continuity`、`State`、`Timelines`、`Logs`、`Traces` 页面
- 快照管理与连续性排障入口

### SDK

当前已具备：

- 基础推理调用
- 世界设置、世界策略访问
- 日志查询、连续性 bundle 读取、调试轨迹读取
- 状态组件、时间线、快照、审批计划等接口封装

## 4. 路线图原则

后续开发遵循以下原则：

1. `Engine` 先落地，再同步 `SDK`、`DevCli`、`Creator`。
2. 可观测性优先于黑盒智能，调试链路必须可追踪。
3. `world_tick` 连续性稳定性优先于新增玩法表层功能。
4. 结构化状态优先于临时拼接文本，能抽组件的尽量抽组件。
5. 文档、示例、镜像文档、打包脚本必须跟随版本同步更新。

## 5. 近期优先级

### P0：推理管线可观测性补齐

目标：让开发者能完整查看引擎在一次推理中的内部链路，而不是只能看到黑盒结果。

重点事项：

- 在 `debug` 和 `review` 模式下，将管线内部关键运行信息全量写入 `logs` 表
- 在 `production` 模式下保留必要的最小化生产日志
- 为 `debug`、`review`、`production` 三种模式定义不同粒度的控制台输出策略
- 为一次请求建立统一 `request_id` 视角，串联日志、轨迹、时间线和结果
- 让 `Creator` 与 `DevCli` 能更容易按 `request_id`、事件名、模式筛查数据

预期结果：

- 能完整复盘单次推理输入、轮次、数据请求、解析结果、动作决策和最终输出
- 管线调优不再依赖猜测，而是依赖结构化证据

### P0：`world_tick` 连续性失真治理

目标：降低世界推进过程中剧情上下文漂移、事实丢失、前后摘要不一致的问题。

重点事项：

- 明确哪些 world tick 产物应该长期持久化，哪些只应作为瞬时推理结果
- 继续把高价值连续性信息收敛到结构化组件中，而不是散落在日志和自由文本里
- 强化 `world_state`、`story_state`、`story_history`、`tick_policy`、`state_snapshot` 之间的职责边界
- 建立连续性 diff、事实核对、异常回溯的排障闭环
- 分析 tick 前上下文构建、tick 中推理、tick 后回写三个阶段的失真来源

预期结果：

- 同一世界多轮推进后，关键设定、事实、剧情主线更稳定
- 开发者可以更快判断问题来自 Prompt、状态回写、还是上下文装配

### P1：状态组件体系扩展

目标：把更多高价值运行态信息从隐式文本中抽出来，放进可管理、可校验、可编辑的组件中。

候选方向：

- 将剧情推进中的阶段性结论抽成新组件
- 将世界级决策约束抽成更清晰的策略组件
- 将中间推理摘要、关键事实集合、冲突线索集合结构化
- 继续增强 Creator 和 DevCli 对这些组件的校验与编辑能力

### P1：工具链同步补齐

目标：避免 Engine 已有能力只能通过 API 间接访问，工具层跟不上。

重点事项：

- `SDK` 为新增 Engine 能力提供稳定接口
- `DevCli` 补齐新增接口的读写、查询、筛选命令
- `Creator` 为新增状态和日志视图提供可视化入口
- 保持四者对同一能力的字段名、事件名、模式名完全一致

### P2：版本与文档治理

目标：控制功能演进后产生的文档漂移和分发信息过时问题。

重点事项：

- 根文档与 `tools/source/docs` 镜像文档同步
- 版本号、示例命令、字段说明、事件名保持一致
- 构建脚本、打包脚本、发布说明随版本更新

## 6. 分阶段推进建议

### 阶段 A：日志与模式分级

- 统一 `debug` / `review` / `production` 的日志策略
- 明确控制台日志与数据库日志的职责边界
- 为关键推理阶段补全事件名和结构化 detail 载荷

### 阶段 B：`world_tick` 持久化审计

- 审查 tick 过程中所有生成数据的落点
- 标记哪些数据应进入节点 / 组件体系
- 标记哪些数据只应进入 `timelines`、`logs`、`traces`

### 阶段 C：连续性组件重构

- 归并重复语义
- 新增缺失的状态组件类型
- 调整状态写入和读取流程

### 阶段 D：排障工作流成型

- 以 `request_id` 为中心串联 continuity、timelines、logs、traces
- 在 Creator 中形成可操作的排障路径
- 在 DevCli 中形成可脚本化的排障命令

### 阶段 E：SDK / DevCli / Creator 同步

- SDK 暴露新接口
- DevCli 补命令和帮助文本
- Creator 补页面和说明文案

### 阶段 F：文档与发布收尾

- 文档对齐
- 示例更新
- 打包脚本与版本号更新

---

## 7. 中期目标

当近期事项稳定后，项目中期目标包括：

- 形成更可靠的 `world_tick` 连续性体系
- 形成面向调试、评审、生产三种环境的标准可观测性方案
- 让 `Creator` 成为连续性排障和世界编辑的一线工具
- 让 `DevCli` 成为适合 AI 协作和自动化脚本的主开发入口
- 让 `SDK` 成为外部游戏集成的稳定边界层

---

## 8. 长期方向

长期来看，GameAgentEngine 需要从“能运行”走向“能稳定开发、能持续调优、能被集成进正式项目”。

长期方向包括：

- 更成熟的世界状态建模规范
- 更稳定的策略审批与自动执行边界
- 更强的推理可解释性与回放能力
- 更完整的多工具协同开发体验
- 更明确的版本兼容与迁移策略

---

## 9. 维护说明

这份路线图不是历史记录，而是当前开发方向说明。每次版本迭代后，至少应同步更新以下内容：

- 当前版本号
- 当前已完成能力基线
- 近期优先级
- 分阶段推进状态
- 与 `Engine` 实际实现不一致的工具或文档项

建议在每次发布 `minor` 版本时复查一次本文件。
